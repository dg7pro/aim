Overview

This feature lets you take a schematic backup of your database, this is recommended from time to time in case you suffer a server/database crash. You should not totally rely on the backup option here, so a further backup with your database administration software is recommended.

On loading the backup page, your database tables are shown along with the current table sizes and an estimated total backup size. If you find the backup is taking too long within this software, you should revert to server options.

You should perform a backup at least once a week. If you are using the e-mail option, e-mail should NOT exist on the same server as your database. If your server crashes, this could also wipe out the e-mail address where your backups are. Use an e-mail address or addresses hosted elsewhere.

Finally, you can set the backup to run automatically if you wish via a cron job/tab. See below.

Download to Hard Drive

Do you wish to download a copy of this backup to your hard drive? This envokes the save-as dialogue box. If not saving, you should enter at least 1 e-mail address to send the backup to.

Compress Backup File

Do you wish to compress the backup file? This is recommended for large databases.

Copy as Attachment to E-Mail Address(es)

Do you want to copy the backup to other e-mail accounts? E-mail accounts should not be located on the same server. Add multiple e-mails if required separated with a comma. Example:

email1@mysite.co.uk,email2@mysite.co.uk

SMTP must be enabled to send attachments. If its not this will be ignored.

Backup as Cron Job/Tab

If you prefer to have the backup generated automatically via a cron job/tab (automated task), follow these instructions:

1. Make sure you have read/write permissions on the following folder. On linux chmod to 755 or 777, depending on which is supported by your host. If you have enabled payment gateway response logging, this may already have the correct permissions:

backups/

2. Set up a cron job/tab in your server admin to point to the following file in your helpdesk installation:

db-backup.php

If you aren`t sure how to do this contact your host. Note that not all hosts support cron job/tabs. Ideally the cron should run when your website is less busy, so early hours of the morning, maybe 3 or 4am.

Some of the following may work as commands, but again, contact your host to check.

/usr/local/bin/php /home/admin/domains/yourdomain.co.uk/public_html/helpdesk/db-backup.php
php q /home/admin/domains/yourdomain.co.uk/public_html/helpdesk/db-backup.php
/usr/local/bin/php q /home/admin/domains/yourdomain.co.uk/public_html/helpdesk/db-backup.php
cmd /k C:\Windows\php\php.exe "C:\website\db-backup.php" (Windows)
3. The default is for backups to save in the 'backups' folder. If you prefer to have them e-mailed, edit the following constant in the 'control/user-defined/defined2.inc.php' file:

BACKUP_CRON_EMAILS

You can have multiple addresses separated with a comma. Note that e-mail addresses should NOT exist on the same server as your database. If the server/database crashes, the e-mails may be lost. Examples:

define('BACKUP_CRON_EMAILS', 'email@mysite.co.uk');
define('BACKUP_CRON_EMAILS', 'email@mysite.co.uk,email2@mysite.co.uk');

Attachments sent and backups saved via the auto cron are Gzip compressed.
